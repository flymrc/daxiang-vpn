## 初步观察(均为待验证假设,非结论)

1. **出口"下载"的瓶颈方向是手机上行,不是下行。** 出口下载数据流向 `公网→手机→Hub`,走蜂窝上行 + WireGuard。实测原始上行 ~5.5、经 WG ~2.7,与出口实测 2–5 吻合。⇒ fast.com(下行)快 ≠ 出口快。**待验证**:上行样本太少、code=100 粗测。
2. **WireGuard 似乎把上行砍了约一半(5.5→2.7)。** 若成立,是可优化项(MTU/分片/Android WG 发包)。**待验证**:仅 1 次对照,需多次重复、排除同期上行波动。
3. **同机同地 UQ 下行 ~50 vs Rakuten 下行 ~6,且 SNR 都低、Rakuten 设置全对、重选无效。** 倾向于"卡×机型"适配/认证差异(此 Moto 非 Rakuten 认证机型),而非可调设置。**待验证**:未做同时刻受控对照,未排除 Rakuten 该时段该位置的网络侧因素;iPhone 数据非同时采集。
4. 我(助手)先前"慢是 SNR/小区拥塞"的判断**已被推翻**(同样低 SNR 下 UQ 能 50)。记录在案以免重复该错误。

## 测量方法的坑(影响数据可信度)

- **fast.com = 多条并行流;curl 默认 = 单流。** 高 RTT/丢包的移动链路上两者差距可达数倍,直接对比会误导。
- **`adb shell su -c "curl ... -w ..."` 的 `-w` 输出常被打成一长串 0(乱码)**,数字不可信;改用控制面 `ssh root@10.66.0.101 'curl ...'` 输出干净。
- **重选小区(飞行模式)结果随机**,单次重选的好坏不代表趋势。
- **librespeed 上行测 code=100**,为粗测,非严格基准。
- 单位:curl `speed_download/upload` 为字节/秒,Mbps = ×8/1e6。

## 要下结论还需补的实验

- 固定卡 + 固定位置,**同一时段多次**测,取分布(中位数/方差),而非单点。
- **同时刻**用 iPhone 与本安卓测同一张卡(受控对照),才能分离"机型"与"网络/时段"。
- 上行:多次、用严格 iperf3/受控 sink 重测原始上行 vs 经 WG,确认 WG 损耗是否稳定存在。
- 出口:统一用多流测聚合吞吐,与单流分开记录。
- 若怀疑 Rakuten 机型适配:换一台 Rakuten 认证安卓机做对照。
