# 出口方案选型

## 目标

本文定义大象 VPN 的出口节点选型策略，回答以下问题：

- 最终出口设备是否必须是 Mac。
- 多个手机卡出口应优先选择哪类硬件。
- 在成本、稳定性、运维复杂度之间如何取舍。
- 当前阶段最适合的 MVP 方案是什么。

本文结论适用于以下目标场景：

- Hub 部署在公网 VPS。
- 多个出口节点主动连接 Hub。
- 出口节点提供真实的住宅或手机网络公网 IP。
- 中国侧客户端通过 Hub 使用指定出口节点上网。

## 结论

当前推荐的出口路线如下：

1. MVP 阶段优先使用便宜安卓机做出口节点。
2. 小规模稳定运行阶段优先使用 OpenWrt 4G 路由器或 OpenWrt 路由器加 4G modem。
3. 大规模、长期、低人工运维阶段再升级到工业蜂窝路由器。

不推荐把 Mac 作为长期最终出口形态。Mac 适合作为临时验证设备，但不适合承载大规模、长期在线、低干预的手机卡出口池。

## 设计原则

出口节点选型按以下原则排序：

- 单节点成本低，允许故障后直接替换。
- 节点可以主动连接 Hub，不依赖公网入站。
- 节点掉线后容易自动恢复。
- 节点扩容时复制成本低。
- 节点状态可观测，能上报公网 IP、握手、延迟、可用性。

## 候选方案

### 方案 A：Mac 加手机热点

拓扑：

```text
iPhone / Android
    |
    | 热点或 USB 共享
    v
Mac
    |
    | WireGuard / 代理客户端
    v
Hub
```

优点：

- 上手快。
- 适合最早期验证链路。
- 适合调试客户端、WireGuard、NAT 行为。

缺点：

- 不是长期生产形态。
- 受系统升级、睡眠、USB 识别、热点重连影响较大。
- 单机成本高于低端安卓机。
- 做多节点时复制成本和人工干预成本都高。

结论：

- 仅用于临时验证。
- 不作为正式出口池推荐方案。

### 方案 B：Root 安卓机直连 Hub

拓扑：

```text
安卓机 + SIM
    |
    | 本机客户端主动连 Hub
    v
Hub
```

适用前提：

- 安卓机可 root，或至少能稳定运行常驻后台进程。
- 客户端程序可以运行在 Android 上。
- 可以接受消费级设备的发热、电池、系统省电策略带来的额外维护。

优点：

- 通常是最低成本方案。
- 每台设备天然就是一个独立手机卡出口节点。
- 不需要 Mac 中转，链路更短。
- 容易做“一机一卡一出口”。
- 公网出口 IP 就是该手机卡的运营商 IP。

缺点：

- 长期运行稳定性受手机厂商系统策略影响很大。
- 发热、电池老化、充电稳定性会成为长期问题。
- 批量管理不如路由器方案整齐。
- 若客户端需要复杂网络能力，Android 端实现和调试成本可能上升。

建议用途：

- 最低成本 MVP。
- 小批量验证。
- 验证 Hub、客户端、出口分配逻辑。

### 方案 C：OpenWrt 4G 路由器

拓扑：

```text
OpenWrt 4G 路由器 + SIM
    |
    | WireGuard / NAT / 策略路由
    v
Hub
```

也可以是：

```text
OpenWrt 路由器
    |
    | USB
    v
4G/5G modem
```

优点：

- 更像正式网络设备。
- 更适合长期在线。
- 更适合承载 NAT、WireGuard、策略路由、多 WAN、自动切换。
- 批量复制和远程维护优于手机。

缺点：

- 成本通常高于便宜安卓机。
- 设备兼容性要重点验证，包括 modem、频段、QMI/MBIM、驱动和固件。
- 某些“基于 OpenWrt”的厂商固件不等于官方原生 OpenWrt，升级和兼容性要单独评估。

建议用途：

- 已经完成 MVP 验证。
- 开始追求稳定在线和标准化运维。
- 需要多线路切换、自动恢复、较少人工干预。

### 方案 D：工业蜂窝路由器

拓扑：

```text
工业蜂窝路由器 + SIM/eSIM
    |
    | 蜂窝 WAN / WireGuard / VPN / 远程管理
    v
Hub
```

优点：

- 稳定性最好。
- 远程管理、告警、故障转移能力强。
- 适合 24x7 长期运行。
- 更适合做正式出口池。

缺点：

- 成本最高。
- 采购和配置门槛更高。

建议用途：

- 节点数量开始增加。
- 人工维护成本已高于硬件成本。
- 业务需要更高 SLA。

## 方案对比

| 方案 | 单节点成本 | 稳定性 | 运维复杂度 | 是否适合作为最终形态 | 当前推荐度 |
| --- | --- | --- | --- | --- | --- |
| Mac + 手机热点 | 高 | 低到中 | 高 | 否 | 低 |
| Root 安卓机 | 低 | 中 | 中 | 可作为过渡形态 | 高 |
| OpenWrt 4G 路由器 | 中 | 中到高 | 中 | 是 | 高 |
| 工业蜂窝路由器 | 高 | 高 | 低到中 | 是 | 中 |

## 成本判断

大方向上，成本排序通常如下：

```text
二手/低端安卓机
  < OpenWrt 路由器 + 4G modem
  < 成品 4G/OpenWrt 路由器
  < 工业蜂窝路由器
```

因此，如果目标是最低成本跑通多手机卡出口，安卓机通常仍然是第一选择。

只有在以下情况下，OpenWrt 路由器可能比安卓机更划算：

- 已经不需要节点本机运行复杂业务，只需要稳定提供出口。
- 团队更擅长网络设备运维，而不是 Android root 和常驻进程维护。
- 希望后续统一使用 OpenWrt 的配置、监控、故障恢复模型。

## 推荐演进路线

### P1：安卓机验证

建议配置：

- 1 到 2 台便宜安卓机。
- 每台设备 1 张 SIM。
- 每台设备作为一个独立出口节点主动连接 Hub。

目标：

- 验证手机客户端或出口客户端是否稳定。
- 验证 Hub 的节点注册、健康检查、出口分配。
- 验证公网 IP、链路延迟、掉线重连。

验收标准：

- 节点可连续在线。
- 断网后可自动重连。
- 客户端可以按出口节点切换。
- 可以稳定拿到手机网络公网 IP。

### P2：OpenWrt 样机

建议配置：

- 1 台 OpenWrt 4G 路由器，或 1 台 OpenWrt 路由器加 1 个 4G modem。

目标：

- 验证 OpenWrt 作为长期出口节点的可维护性。
- 验证自动重拨、WireGuard、NAT、策略路由和健康检查。
- 对比与安卓机方案的稳定性和运维成本。

验收标准：

- 连续运行时间优于安卓机。
- 远程维护更容易。
- 不依赖人工重启即可恢复常见故障。

### P3：正式出口池

根据 P1 和 P2 的结果选择：

- 若成本敏感，继续扩展安卓机节点。
- 若稳定性优先，扩展 OpenWrt 节点。
- 若业务已形成固定规模，逐步升级到工业蜂窝路由器。

## 不推荐的路径

以下路径不建议作为正式长期架构：

- 多个手机统一接到一台 Mac，再由 Mac 统一转发。
- 长期依赖 iPhone 或 Android 的 Wi-Fi 热点作为主链路。
- 没有散热、供电、保活机制的手机直连方案。

这些做法的问题主要是：

- 故障点集中。
- 热点重连不稳定。
- 维护成本高。
- 扩容不自然。

## 运维要求

无论最终采用安卓机还是 OpenWrt，出口节点都应满足以下要求：

- 可主动连接 Hub。
- 可上报自身名称、角色、当前公网 IP、最近握手、延迟、可用性。
- 可自动重连。
- 可在重启后自动恢复。
- 有统一命名规范，例如 `jp-mobile-01`、`jp-owrt-01`。

建议额外采集：

- 运营商名称。
- SIM 标识。
- 节点温度。
- 最近重连时间。
- 流量用量。

## 当前建议

当前仓库阶段，推荐采用以下决策：

1. 先用便宜安卓机完成真实手机卡出口验证。
2. 同时准备 1 台 OpenWrt 4G 样机做并行评估。
3. 不继续围绕 Mac 设计最终出口架构。
4. 后续将 Hub 的出口管理逻辑统一抽象为“节点”，不区分底层是安卓机还是 OpenWrt。

## 与仓库架构的对应关系

在本仓库中，出口设备应统一视为 `egress` 角色：

- `jp-mobile-01` 可以是安卓机。
- `jp-owrt-01` 可以是 OpenWrt 路由器。
- `jp-router-01` 可以是工业蜂窝路由器。

控制平面不应依赖底层设备类型，而应依赖统一的状态字段，例如：

- `role: egress`
- `egress_type: mobile`
- `carrier: ...`
- `public_ip: ...`
- `enabled: true`
- `healthy: true`

这样后续切换硬件时，不需要推翻 Hub 和客户端设计。
