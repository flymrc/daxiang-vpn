# Windows GUI 客户端优化 TODO

> 目标:减少用户误解、降低排障成本、让客户端更像稳定可交付的桌面软件。

## P0 用户可感知稳定性

- [ ] **第二次打开时激活已有窗口**
  - 当前 0.4.4 已用 Windows Mutex 防止多开,但第二个进程直接退出,用户会感觉“点了没反应”。
  - 目标:第二次启动时通知已有实例 `show + unminimize + focus`。
  - 可选实现:本地命名管道 / loopback IPC / Tauri single-instance 插件评估。

- [x] **IPv4 / IPv6 展示如实区分可用性**（0.4.5 已完成）
  - 当前 GUI 分开展示「出口 IPv6」「出口 IPv4」。
  - 目标:Hub 不作为出口兜底;IPv4 烂就显示烂,断了就显示断。
  - 建议文案:
    - `出口 IPv6`: `<Rakuten IPv6>`
    - `出口 IPv4`: `<Rakuten IPv4>` 或 `不可用`
  - 0.4.5 行为:若探测结果等于 Hub endpoint IP,客户端不把它当作出口 IPv4 展示;若 IPv4 探测完成但无手机 IPv4,显示 `不可用`。
  - 界面保持短,不要放长解释。

- [ ] **连接后后台轻量可用性检测**
  - 不只判断 `127.0.0.1:7890` 端口可达。
  - 连接成功后后台检测:
    - IPv6 出口 endpoint。
    - 普通 HTTPS 双栈站。
    - v4-only endpoint 失败应作为 IPv4 出口异常信息,不能改由 Hub 兜底。
  - 失败不阻断连接,只更新内部状态或轻提示。

## P1 排障与恢复

- [x] **复制诊断信息**（0.4.5 已完成基础版）
  - 主界面或托盘增加“复制诊断信息”。
  - 内容应包含:
    - GUI 版本。
    - sidecar 版本/hash。
    - 本地代理地址与端口可达性。
    - `status --json --no-ip-check` 输出。
    - 最近一次 IPv4/IPv6 探测结果与时间。
    - Windows 系统代理是否指向本地代理。
  - 严禁包含 token、私钥、完整 WireGuard 配置。
  - 0.4.5 已加入主界面“复制诊断”,包含 GUI 版本、连接状态、当前 status JSON、最近一次 IPv4/IPv6 探测结果与时间;sidecar hash、系统代理指向检查后续再补。

- [ ] **系统代理恢复失败可诊断**
  - 当前已有 backup/restore 兜底。
  - 目标:恢复失败时写入清晰日志,并在 GUI 给出可操作提示。
  - 覆盖场景:
    - 断开时恢复失败。
    - 托盘退出时恢复失败。
    - 崩溃后下次启动恢复失败。

- [ ] **端口占用提示优化**
  - 当前 `7890` 被占用时会报错。
  - 目标:
    - 判断占用者是否像自己的旧 `zhvpn.exe __engine`。
    - 是旧进程:尝试清理或给出“发现旧实例”提示。
    - 非自身进程:提示端口被占用,给出关闭占用程序或换端口建议。
  - 暂不建议默认自动换端口,避免系统代理与用户手动配置失配。

## P2 产品化

- [ ] **Windows Credential Manager 保存授权**
  - 目标:token 不再依赖普通配置文件或 WebView localStorage。
  - GUI 登录态与 CLI 配置需要设计兼容策略。
  - 注意迁移旧 token 与登出清理。

- [ ] **开机自启 + 自动重连**
  - 默认关闭,设置中显式开启。
  - 恢复上次连接参数:
    - 全局代理。
    - 高性能模式。
  - 启动失败不弹无限窗口,只在托盘/主界面提示。

- [ ] **代码签名**
  - 目标:降低 Windows SmartScreen / 杀软 / 客户心理门槛。
  - 需要确定证书、签名流程、CI 或本机构建流程。

## P3 后续架构增强

- [ ] **状态模型分层**
  - 内部状态拆为:
    - 本地代理。
    - WireGuard 到 Hub。
    - Android 出口在线。
    - 公网 IP 观察。
  - UI 仍保持简单,但诊断信息和错误提示更精准。

- [ ] **PID 文件残留自愈**
  - 0.4.4 已有操作级 Mutex,但 PID 文件损坏/复用仍可增强。
  - 目标:
    - PID 存在但非当前 `ZHVPN_HOME` engine 时不误杀。
    - PID 文件缺失但端口仍被自身旧进程监听时可识别。
    - `start/stop/status` 给出一致判断。

- [ ] **真正完整 TUN 全局模式**
  - 现有「高性能模式」不是完整全局 TUN。
  - 后续需单独设计:
    - tun inbound。
    - auto_route / strict_route。
    - DNS 与 IPv6 泄漏处理。
    - 断开后路由恢复。

## 建议顺序

1. 第二次打开时激活已有窗口。
2. 连接后后台轻量可用性检测。
3. 系统代理恢复失败可诊断。
4. 端口占用提示优化。
