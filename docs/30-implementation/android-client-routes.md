# Android 客户端三条路线

## 目标

本文用来比较 Android 出口节点客户端的三条实现路线：

1. `Go 二进制直接在 root 安卓上跑`
2. `Go 核心 + Kotlin/Java 外壳`
3. `纯 Kotlin/Java Android App`

本文只讨论“安卓作为出口节点”的场景，不讨论中国侧桌面用户客户端。

## 背景

当前项目的目标不是先做一个漂亮的 Android UI，而是先让 rooted 安卓手机尽快成为可用的 `egress` 节点：

- 主动连接 Hub
- 暴露 WireGuard 内网代理端口
- 用手机卡网络作为公网出口
- 可复制到多台设备

因此，路线选择的优先级是：

- 尽快跑通
- 最大化复用现有代码
- 降低今晚联机验证难度
- 给后续保活、前台服务、运维能力留空间

## 路线一：Go 二进制直接运行

### 形态

```text
Windows 开发机
    |
    | go build GOOS=android GOARCH=arm64
    v
单个 Android ELF 二进制
    |
    | adb push + su + shell
    v
root 安卓手机
```

### 适合做什么

- 第一版出口节点守护进程
- WireGuard / sing-box / 代理 / 健康检查
- 快速验证 rooted 安卓机是否能稳定当出口

### 优点

- 复用现有 Go 代码最多
- 复用现有 `sing-box` 集成最多
- 不需要先搭 Android App 工程
- 最适合今晚这种“先推到机器上跑起来”的节奏
- 调试路径短，问题边界清楚

### 缺点

- 没有原生 Android UI
- 没有现成的前台服务、通知、开机自启
- 保活和系统集成需要后续补
- 分发方式更像运维工具，不像面向用户的 App

### 当前适配度

对本项目而言，这条路线是当前最适合的第一步。

原因：

- 仓库已有 Go 客户端和 Go 侧配置模型
- 现有核心网络逻辑已经围绕 Go 展开
- rooted 安卓机允许通过 shell 先跑二进制

### 推荐阶段

- `现在`
- MVP 验证阶段
- root 机联机调试阶段

## 路线二：Go 核心 + Kotlin/Java 外壳

### 形态

```text
Kotlin/Java Android App
    |
    | JNI / gomobile / 子进程管理
    v
Go 网络核心
    |
    v
WireGuard / sing-box / 出口守护进程
```

### 适合做什么

- 已经证明 Go 出口核心可跑后
- 需要 Android 前台服务、通知、权限控制
- 需要更像正式产品的安装、启动、停止、状态展示

### 优点

- 保留 Go 网络核心复用优势
- 同时获得 Android 原生壳层能力
- 更容易做：
  - 前台服务
  - 开机自启
  - 日志查看
  - 状态页
  - 一键启动/停止
- 后续如果不再只面向 root 机，也更容易演进

### 缺点

- 工程复杂度明显高于路线一
- 会引入 Go 与 Android 壳层之间的边界问题
- 调试链路更长
- 第一版开发速度会慢一些

### 当前适配度

这条路线很适合做“第二阶段产品化”，但不适合抢在今晚之前做第一版。

### 推荐阶段

- Go 守护进程验证成功之后
- 需要稳定保活和更好操作体验时
- 需要让非开发人员也能上手维护节点时

## 路线三：纯 Kotlin/Java Android App

### 形态

```text
Kotlin/Java Android App
    |
    v
所有网络核心、生命周期、配置、守护逻辑都在 Android 侧实现
```

### 适合做什么

- 从一开始就计划做完整 Android 产品
- 团队 Android 工程能力很强
- 不依赖现有 Go 网络核心

### 优点

- Android 平台集成最自然
- UI、服务、权限、通知、系统生命周期最好处理
- 从 Android 研发视角最标准

### 缺点

- 复用现有 Go 代码最少
- 需要重新处理网络核心实现或重新封装已有能力
- 容易把现在的“先跑通出口节点”目标拖慢
- 第一版风险最高、周期最长

### 当前适配度

对当前项目不合适。

当前项目已经有：

- Go 客户端基础
- Go 配置模型
- Go 内嵌 `sing-box`

这意味着纯 Kotlin/Java 路线会放弃现有积累，换来一个更慢的起点。

### 推荐阶段

- 只有在未来明确决定“安卓是主产品平台”，且愿意重构网络层时才考虑

## 三条路线对比

| 维度 | 路线一：Go 直跑 | 路线二：Go + Kotlin 外壳 | 路线三：纯 Kotlin/Java |
| --- | --- | --- | --- |
| 跑通速度 | 最高 | 中 | 最低 |
| 代码复用 | 最高 | 高 | 低 |
| Android 原生体验 | 低 | 高 | 高 |
| 工程复杂度 | 低 | 中到高 | 高 |
| 今晚可落地性 | 最高 | 低 | 很低 |
| 长期产品化能力 | 中 | 高 | 高 |
| 当前推荐度 | 最高 | 中 | 低 |

## 当前建议

当前推荐明确如下：

1. 先走 `路线一：Go 二进制直接运行`
2. 跑通 rooted 安卓出口节点后，再补 `路线二：Go + Kotlin/Java 外壳`
3. 暂不考虑 `路线三：纯 Kotlin/Java Android App`

这不是说路线三做不到，而是说它不符合当前阶段最重要的目标：

- 先让节点上线
- 先验证手机卡出口稳定性
- 先复用现有仓库能力

## 与当前仓库的对应关系

当前仓库已经为路线一准备了第一版入口：

```text
egress/proxy
```

相关文档：

- [Android 出口节点客户端](./ANDROID_EGRESS_AGENT.md)

因此，当前实际开发顺序应是：

1. 继续完善 `dxandroid-egress`
2. 在 rooted 安卓机上完成联机验证
3. 基于 Android 出口到 Hub 的 25 秒 WireGuard keepalive 继续补自恢复
4. 需要时再加 Android 外壳

## 一句话结论

如果目标是“今晚先把 rooted 安卓手机变成可用出口节点”，选 `Go 直跑`。

如果目标是“后面做成更像产品的 Android 节点管理 App”，再演进到 `Go + Kotlin 外壳`。

如果目标是“完全重做为 Android 原生产品”，才考虑 `纯 Kotlin/Java`。
