# zhvpn.exe 本地单例实现计划

## 背景

当前客户端已经实现“默认使用场景下的单例”：

- 同一个 `ZHVPN_HOME` / `%LOCALAPPDATA%\ZonghengVPN` 下维护一个 PID 文件。
- 默认监听一个本地代理端口：`127.0.0.1:7890`。
- `start` 会读取 Hub 最新 bootstrap 配置。
- 运行配置未变化时，`start` 返回“已在运行”。
- 运行配置变化时，`start` 自动停止旧代理并启动新代理。
- 如果本地端口被其它进程占用，`start` 报错，不再误判为自身已运行。

2026-06-11 起，GUI 主程序已用 Windows 命名 Mutex 做应用级单例，重复打开第二个 `zhvpn-desktop.exe` 会直接退出；CLI 的同一实例操作也已补命名 Mutex。下面保留边界与后续增强项。

## P0：同一实例并发 start 互斥（已完成）

### 问题

两个 `zhvpn.exe start` 在同一瞬间执行时，可能同时通过“未运行/端口未监听”的检查，然后都尝试写运行配置、启动引擎。

这是当前最值得优先处理的单例边界，因为它会发生在同一个默认实例内，且用户无感。

### 目标

同一个 `ZHVPN_HOME` 下，同一时间只允许一个 `start` / `stop` / 配置切换流程进入临界区。

### 方案

Windows 侧使用命名 Mutex。

当前实现：

```text
Local\ZonghengVPN-{first 8 bytes of sha256(ZHVPN_HOME) as hex}
```

原因：

- `Local\` 限定当前登录会话，适合桌面客户端。
- 用 `ZHVPN_HOME` hash 区分不同实例目录。
- 不把完整路径放入 mutex 名，避免非法字符和路径泄露。

### 影响命令

- `start`
- `stop`
- `login`
- `import`
- `logout`
- 隐藏命令 `__engine` 不需要拿锁，它是被管理的后台进程

### 验收

1. 并发执行两个 `zhvpn.exe start`。
2. 最终只有一个后台代理进程。
3. `127.0.0.1:7890` 只有一个监听 PID。
4. PID 文件与监听 PID 一致。
5. 第二个命令应等待第一个完成，或明确提示“另一个操作正在进行”。

## P1：PID 文件损坏 / 极端残留的自愈

### 问题

PID 文件可能出现以下异常：

- 文件不存在，但端口仍被旧进程监听。
- 文件存在，但 PID 已不存在。
- 文件存在，PID 被系统复用成其它进程。
- PID 指向 zhvpn 进程，但不是当前 `ZHVPN_HOME` 的后台引擎。

当前代码已经能处理部分情况，但还不够严格。

### 目标

`start` / `stop` / `status` 都能给出可信判断，不被坏 PID 文件误导。

### 方案

扩展运行状态文件，记录：

```text
pid
started_at
exe_path
home
runtime_fingerprint
local_proxy_addr
```

同时增强进程校验：

- PID 存在。
- 进程名是 `zhvpn.exe`。
- 命令行包含 `__engine`。
- 命令行包含当前 `--home <ZHVPN_HOME>`。
- 如果 PID 不可信，删除 PID 文件并继续端口检查。

### 验收

1. 手动写入不存在的 PID，`status` 显示未运行或根据端口判断。
2. 手动写入其它进程 PID，`stop` 不杀错进程。
3. 删除 PID 文件但保留后台代理时，`start` 能识别端口占用并给出明确提示。
4. 正常后台代理仍能被 `stop` 精准停止。

## P2：不同端口多实例策略

### 问题

用户可以执行：

```powershell
zhvpn.exe start --port 7891
```

这可能启动另一套本地代理，绕开默认 `7890` 的单例语义。

### 目标

明确产品策略：是否允许同一 `ZHVPN_HOME` 下多端口多实例。

### 推荐策略

短期建议不允许。

理由：

- 客户端面向普通用户，多实例会增加诊断复杂度。
- 出口切换应该由服务端配置或后续动态调度完成，不靠本地多开。
- 当前 token / 配置模型本质上是“一个客户端一个当前出口”。

### 实现

保留 `--port` 作为端口覆盖，但仍受同一 `ZHVPN_HOME` 单例约束：

- 如果已有实例运行，即使新命令指定不同端口，也视为配置变化。
- 自动重启到新端口，而不是并行启动第二个实例。

### 验收

1. 先启动默认 `7890`。
2. 再执行 `start --port 7891`。
3. 旧 `7890` 被释放。
4. 新 `7891` 监听。
5. 后台仍只有一个 zhvpn engine。

## P3：不同 ZHVPN_HOME 的全局策略

### 问题

用户可以设置不同的 `ZHVPN_HOME`：

```powershell
$env:ZHVPN_HOME="C:\tmp\zhvpn-a"
zhvpn.exe start

$env:ZHVPN_HOME="C:\tmp\zhvpn-b"
zhvpn.exe start --port 7891
```

这会形成多套独立实例。

### 目标

决定是否允许“高级多实例”。

### 推荐策略

保留允许，但文档标注为高级/调试用途。

理由：

- 多 `ZHVPN_HOME` 对开发、测试、灰度有价值。
- 只要端口不同，系统层面可并存。
- 如果要产品级禁止，需要全局 mutex，反而会影响测试和排障。

### 实现

默认不做全局禁止，只做清晰文档：

- 普通用户不需要设置 `ZHVPN_HOME`。
- 多 `ZHVPN_HOME` 视为多实例高级模式。
- 每个 `ZHVPN_HOME` 内仍必须满足 P0/P1/P2 的单例规则。

### 验收

1. 默认 `ZHVPN_HOME` 启动一个实例。
2. 自定义 `ZHVPN_HOME` + 不同端口可启动第二个实例。
3. 两个实例 PID、运行状态、指纹文件互不覆盖。
4. 如果端口冲突，第二个实例启动失败并提示端口占用。

## 建议实施顺序

```text
P0 -> P1 -> P2 -> P3
```

其中：

- P0 是必须做的并发安全。
- P1 是稳定性和防误杀。
- P2 是产品语义收口。
- P3 是高级能力边界，不急于限制。

## GUI 主程序单例

GUI 主程序单独使用一把固定的当前会话 Mutex：

```text
Local\ZonghengVPNDesktop
```

目标是防止用户重复双击打开多个窗口、多个托盘、多个状态轮询。第二个 GUI 进程发现 Mutex 已存在后直接退出；已有窗口保持运行。

## 当前结论

当前版本已经解决：

- 出口配置变化但旧进程仍连接旧出口的问题。
- GUI 主程序重复打开的问题。
- 同一 `ZHVPN_HOME` 下 `start/stop/login/import/logout` 并发进入临界区的问题。

后续重点是 P1：PID 文件损坏、PID 复用、端口残留等极端状态下的更强自愈与防误杀。
